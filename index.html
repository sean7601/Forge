      Security Explainer: Why Forge Apps Are Secure
      Mobile Code is governed under SC-18 in the host ATO boundary

      Forge apps are mobile code, compiled for offline browser execution with no required server, no database, and no app-hosted API surface. That removes by default automated interconnects common to normal web systems.

      Forge adds defense-in-depth controls for offline builds: restrictive security headers, URL/form/link hardening, iframe isolation with a mediated bridge for sensitive operations, and verification artifacts. Hosted builds are different: SharePoint Firepit and Fusion Firepit paths use their host renderer as the security boundary, while Fusion copy/paste builds include Forge's Fusion security wrapper directly in the HTML macro.

      For RMF: treat this under SC-18 Mobile Code and host-system controls. The host browser is the authorized system boundary; the HTML/JS file is managed as mobile code executed inside that boundary. The browser maintains the ATO, not the html file application.

      Mobile Code Categories
      • Offline HTML/JavaScript in an approved, STIG-compliant browser with network egress blocked is treated as Category 3 (generally acceptable).
      • Category 3 is acceptable specifically because execution is constrained by the browser and does not permit unmediated host-service access.
      • Category 2 examples (such as VBA/Excel macros) need additional safeguards and tighter enterprise policy controls.
      How the Internet Model Maps to Forge
      1. Request: Your computer asks for a website.
      2. Delivery: A server sends HTML/CSS/JS files.
      3. Execution: Your browser runs those files locally.

      In normal web use, your computer asks a server for a site, the server sends HTML/CSS/JS files, and your browser executes those files locally. If you go to facebook.com on your government computer, an unknown server is sending you html/js/css files that are then executing on your government-supplied computer. We accept this because html/css/js is category 3 mobile code run in a secure browser.

      Forge follows that same browser execution model, but the files are already on your computer instead of being pulled at runtime from a server you may not trust.

      That means the security emphasis is on local code control and verification in the approved browser boundary, rather than trusting unknown remote server-delivered content.

      ATO Boundary: Why This Is Different From a Traditional Web App

      Traditional web-app ATOs focus heavily on server-side and platform concerns: web server hardening, database security, API authz/authn, transport protections, cloud boundary controls, patching, secrets management, and backend monitoring.

      Forge offline apps intentionally do not include that architecture. There is no required server runtime, no app database service, and no internet-facing endpoint for the app itself. That removes large portions of conventional attack surface and associated control burden.

      What remains is mobile-code governance in the authorized host environment: SC-18 definitions, host/browser baseline, and local execution safeguards.

      SharePoint Enclave Pattern (SPFx Hosting)

      When your program treats its SharePoint tenant/site as an approved enterprise enclave, Forge can compile for Firepit without adding Forge's offline security headers to the HTML artifact.

      In the SharePoint path, Forge ships a clean HTML file and Firepit, the SPFx host, provides the sandboxed wrapper and optional lockdown CSP inside the SharePoint page. This avoids stacking Forge's offline wrapper on top of Firepit's host controls.

      SharePoint API traffic stays inside the SharePoint tenant/site boundary and uses the user's existing SharePoint identity and permissions.

      Result: SharePoint-hosted apps run through Firepit's host security model, while offline Forge builds continue to receive Forge's security headers and isolated runtime wrapper.

      Fusion Enclave Pattern (Confluence HTML Macro Hosting)

      When Fusion is the approved collaboration enclave, Forge treats the Fusion page, HTML macro, uploaded attachments, page properties, and current user's Confluence session as the host boundary.

      Copy/Paste Deployment places the secured Forge output directly into the Fusion HTML macro. That output includes Forge's Fusion wrapper, sandboxed iframe execution, URL/form/link hardening, and a parent bridge that allows Confluence page-property persistence only through the current user's Fusion session.

      Firepit Deployment stores only the small fusion-firepit.html renderer in the HTML macro. The larger app is uploaded as a Fusion page file or attachment and is loaded by the renderer into a sandboxed iframe. In this path, the uploaded app skips Forge's embedded wrapper because the Firepit renderer supplies the host security boundary.

      Result: Fusion apps do not introduce a separate backend service. Data persistence and file retrieval stay in the Fusion/Confluence page context, subject to the user's existing Fusion permissions and the page's configured access controls.

      Why Category 3 Offline Browser Apps Are Safer Than Category 2 Excel/VBA
      Security Dimension Offline Forge App (Category 3 pattern) Excel/VBA Macro (Category 2 pattern)
      System access model Browser-constrained execution with mediated access to privileged host APIs. Richer host interaction potential and broader automation capability inside office/runtime context.
      Default network posture Offline-first with CSP/guard controls designed to suppress outbound calls and URL leakage. Often relies on enterprise macro policy and trust/signing controls; network interactions can still exist by design.
      Exfil pathway surface Browser/web exfil routes are blocked or sanitized at compile/runtime layers. Macro-enabled document workflows can expose more channels if host controls are weak or bypassed.
      Deployment architecture risk No required web server/database tier for app operation. Typically part of broader office/email/share workflows with larger enterprise integration surface.
      Assurance artifacts Forge provides built-in security review/testing outputs and deterministic packaging artifacts. Assurance often depends on external scanning/governance tooling and organizational macro controls.
      What Forge Actually Does to Secure Apps (Control Inventory)
      1. Offline restrictive CSP injection: default deny model with default-src 'none', connect-src 'none' by default (or explicit API allowlist), form-action 'none', frame-src 'none', object-src 'none', and manifest-src 'none'. Hosted Firepit builds skip this Forge header layer so the host renderer can own the security wrapper.
      2. Offline header hardening: adds Referrer-Policy: no-referrer, X-Content-Type-Options: nosniff, Permissions-Policy, and DNS prefetch suppression. SharePoint-hosted and Fusion Firepit pages handle their side of the security header/wrapper behavior instead.
      3. Link-hint neutralization: runtime strips/neutralizes risky <link rel> hints with browser-dependent CSP behavior (for example prefetch/prerender-like paths).
      4. URL/query hardening: runtime sanitizes navigation targets and strips query strings by default; only approved mail compose params (subject, body, cc, bcc) are retained for mailto:. In SharePoint compatibility mode, query params are allowed for the configured SharePoint origin only.
      5. Form exfil controls: sanitizes form actions and blocks GET submissions that leak data in URLs.
      6. Embedded content controls: sanitizes iframe/frame/embed src, object data, and srcdoc; strips meta refresh navigation patterns.
      7. Dynamic DOM enforcement: MutationObserver continually re-sanitizes newly inserted or mutated attributes.
      8. Isolated execution shell: compiled app runs inside sandboxed iframe; sensitive actions are mediated by parent-shell bridge, not directly by untrusted app script.
      9. Gesture-gated capability bridge: directory/file picker and related host actions require explicit user approval events.
      10. First API-call approval gate: for non-SharePoint allowlisted API origins, the host shell prompts for explicit user permission before outbound API calls are allowed.
      11. SharePoint Firepit hosting mode: Ship for SharePoint outputs HTML without Forge-added security headers; Firepit loads that file from SharePoint and provides the SPFx-hosted sandbox/wrapper.
      12. SharePoint context seeding: Firepit seeds _spPageContextInfo into the hosted app so SharePoint context detection remains stable in iframe deployments.
      13. Fusion copy/paste hosting mode: Ship for Fusion copy/paste includes Forge's Fusion wrapper inside the HTML macro, with sandboxed app execution and Confluence page-property persistence mediated through the parent bridge.
      14. Fusion Firepit hosting mode: Ship for Fusion Firepit outputs an uploaded app file without the embedded wrapper; the fusion-firepit.html renderer in the Fusion HTML macro loads the app and supplies the sandboxed host boundary.
      15. Integrity and evidence outputs: manifest/hashes and security testing workflows support auditability and reproducibility.

      SharePoint Deploy Guide: Firepit, Legacy + Lists
      SharePoint Hosting Options

      Firepit is the SharePoint/SPFx host for Flank Speed Forge apps. It loads a Forge-built HTML file from the same SharePoint site and runs it inside Firepit's sandboxed security wrapper.

      Legacy IntelShare does not use Firepit. Ship or deploy the app as a clean .aspx file and place it directly in the target site's SitePages folder on intelshare.intelink.sgov.gov or intelshare.intelink.gov.

      1. Add Firepit To The Site
      1
      Open the target SharePoint site home page

      Start from the home page of the site where the app will live.

      2
      Add the Firepit client-side solution

      Select Create, choose App, then add Firepit client-side solution to the site. This is a one-time setup per site.

      If Firepit is already added to the site, skip this step. SharePoint may not let the same app be added twice.
      2. Ship And Upload The HTML App
      A
      Ship for SharePoint or Legacy IntelShare

      In Forge, open Ship and choose Ship for SharePoint for Firepit or Ship for Legacy IntelShare for direct .aspx SitePages hosting. Both outputs skip the offline security headers/wrapper.

      B
      Use the current file

      Forge names the latest SharePoint build like app-folder - Only Secure in FS Sharepoint-current.html. When a new version is shipped, the old current file is archived with its version number and the newest build becomes the -current file.

      C
      Upload and copy the link

      Upload the -current HTML file to a SharePoint document library such as Documents, then copy the link to that uploaded HTML file.

      3. Create The Firepit Page
      1
      Create a new page

      Return to the site home page, create a new page, and choose the Firepit page template. You may need to scroll through the template options to find it.

      2
      Point Firepit at the uploaded HTML

      In the Firepit configuration panel, name the page/site and paste the copied HTML file link into HTML File URL. If the pasted URL includes a trailing ?csf=... query string and your Firepit version does not remove it automatically, delete that query string before saving.

      3
      Choose Firepit options and save

      Enable Full screen mode if the app should fill the page. Use Lockdown mode only when the app should run without connecting to SharePoint APIs. Save the page when the configuration is complete.

      4. Update An Existing Firepit App

      Ship the app again with Ship for SharePoint, upload the new -current HTML file to the same SharePoint location, and choose Replace when SharePoint asks what to do with the existing file.

      Because the Firepit page points at the same -current file name, the page URL and Firepit configuration stay the same. Refresh the page to load the new app version.

      Legacy IntelShare Direct Page

      Use Ship for Legacy IntelShare or choose Legacy IntelShare SitePages .aspx in the SharePoint deploy panel. Forge produces or uploads a clean .aspx file directly into SitePages.

      This path is for intelshare.intelink.sgov.gov and intelshare.intelink.gov. It skips Firepit entirely, so page behavior depends on legacy SharePoint's page/script handling and the current user's permissions.

      5. Optional: Paste HTML Directly

      Firepit also supports pasting the HTML code directly into the web part configuration instead of pointing at an uploaded file.

      Use this only for small apps or special access cases. SharePoint property values are limited to about 750 KB, but direct-pasted HTML can avoid the "download file" permission behavior that may block users on personal computers even when they can view the site.

      6. Add SharePoint List Integration

      For data persistence, open Prompt Lab, switch to Feature Add, choose SharePoint List Integration, and describe what the app should save. Provide the SharePoint site URL, usually the site root such as https://flankspeed.sharepoint-mil.us/sites/your-site or https://intelshare.intelink.gov/sites/your-site.

      If the list, library, or folder already exists, check I already created the SharePoint list/library/folder and provide the exact details. Otherwise, the generated code can create the SharePoint list automatically the first time the app opens, assuming the current user has permission.

      After AI updates the code, ship for the same SharePoint target again. For Firepit, replace the uploaded -current HTML file and refresh the Firepit page. For legacy IntelShare, replace the -current.aspx page in SitePages.

      7. Optional: Add Live Polling

      Basic SharePoint list integration reads and writes data, but another user's changes may not appear until refresh. For collaborative behavior, use Prompt Lab -> Feature Add -> SharePoint Live Polling after the base list integration works.

      Then ship for SharePoint again, replace the same -current HTML file, and refresh open pages. The app should poll SharePoint and merge remote changes without requiring manual reloads.

      Security Boundary

      Do not rely on front-end passwords, hidden screens, or client-side user checks to protect SharePoint data. Users who have access to the underlying SharePoint list can still open the list directly. Sensitive data must be protected with SharePoint list, library, folder, or item-level permissions.

      System Diagram: How a Firepit App Works
      User Operator opens a SharePoint site page

      The user lands on a normal site page inside the target site.

      ->
      SharePoint Page Page contains the Firepit web part

      Firepit is the SPFx host on the page. It knows which HTML file to render from the site.

      ->
      Firepit Host Firepit loads compiled Forge HTML

      The web part fetches the uploaded HTML from SharePoint and renders it in a sandboxed iframe.

      ->
      SharePoint Backend App reads and writes SharePoint data

      The app uses same-origin SharePoint REST calls to lists and document libraries for persistence.

      End-to-end model: User -> SharePoint Site Page -> Firepit (SPFx host) -> uploaded SharePoint HTML file -> SharePoint lists/libraries. Firepit hosts the app; SharePoint remains the storage and identity boundary.

      Deployment Sequence To Remember
      1. Add the Firepit client-side solution to the SharePoint site.
      2. Use Ship for SharePoint in Forge and upload the -current HTML file to a document library.
      3. Create a new Firepit page and paste the uploaded HTML file URL into HTML File URL.
      4. For updates, replace the same -current HTML file so the Firepit page keeps working without reconfiguration.
      5. Use SharePoint List Integration and optionally SharePoint Live Polling when the app needs shared data.
      Fusion Deployment Guide
      Fusion Wiki HTML Macro

      Use Ship for Fusion from the Ship modal. Choose Copy/Paste Deployment for the quickest path when the shipped HTML is small enough for the Fusion HTML macro, or choose Firepit Deployment for larger apps because the macro only hosts the renderer and the app itself is loaded from an uploaded file. Use the Full screen toggle with either path.

      Copy/Paste Deployment

      This is the fastest deployment path, but it is limited by how much HTML Fusion will accept inside one HTML macro. Use it for smaller apps and quick updates.

      1. Choose Ship for Fusion, set the Full screen toggle, then choose Copy/Paste Deployment.
      2. Create a space in Fusion.
      3. Press Create at the top to create a page.
      4. Press the + sign.
      5. Select Other macros.
      6. Search for html.
      7. Select HTML.
      8. Press Insert.
      9. Paste the copied shipped code into the HTML area.
      Firepit Deployment

      This path is not constrained by the HTML macro's pasted-code size limit because Fusion only stores the small Firepit renderer in the macro. The full app is uploaded as a file and loaded by URL.

      1. Choose Ship for Fusion, set the Full screen toggle, then choose Firepit Deployment.
      2. Upload the saved -current.html app file from Shipped Apps to the Fusion page file list or attachments area.
      3. Copy the uploaded HTML file link.
      4. Add an HTML macro to the Fusion page and paste the copied fusion-firepit.html renderer code into it.
      5. Publish the page, paste the uploaded HTML file link into HTML file link, and press Render App. The renderer saves that link in the page properties for future page loads.
      Updating A Fusion Page

      For copy/paste deployment, ship again, edit the HTML macro, replace the old code, and publish. For Firepit deployment, ship again and replace the uploaded -current.html file in Fusion; the renderer keeps pointing at the same file link.

      Security Behavior

      Copy/Paste Deployment includes Forge's Fusion security wrapper. Firepit Deployment saves the uploaded app without that wrapper because the Firepit renderer supplies the host security boundary.
